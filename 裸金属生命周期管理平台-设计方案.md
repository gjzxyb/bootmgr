# 统一裸金属生命周期管理平台设计方案

## 1. 项目概述

**项目名称**: 统一裸金属生命周期管理平台 (Unified Bare Metal Lifecycle Platform)

**设计目标**: 融合 Tinkerbell、HrnrKitOS 和 iVentoy 的关键能力，建设一个面向数据中心和私有云场景的裸金属服务器全生命周期自动化管理平台，覆盖设备发现、资产管理、网络启动、系统部署、运行监控、带外管理、运维工具和退役审计。

**核心价值**:

- 降低裸金属部署和运维复杂度，减少人工装机、人工巡检和人工记录
- 提供从硬件发现到退役报废的完整生命周期闭环
- 统一 PXE/iPXE、BMC、CMDB、镜像、工作流、监控和审计入口
- 支持传统 x86 数据中心，并为 ARM64 和信创操作系统预留扩展能力
- 在 MVP 阶段优先交付可验证、可部署、可演示的核心链路

## 2. 建设范围与非目标

### 2.1 建设范围

平台建设范围分为四层：

1. **基础启动与部署层**: DHCP/ProxyDHCP、TFTP、HTTP、iPXE、ISO 镜像管理、自动化安装模板、元数据服务。
2. **资产与生命周期层**: 设备发现、硬件采集、CMDB、状态流转、标签、机柜位置、租户归属和审计记录。
3. **运行运维层**: 监控采集、告警、自愈策略、BMC 电源控制、WebSSH、批量脚本、日志采集。
4. **平台治理层**: 认证鉴权、RBAC、多租户、操作审计、备份恢复、API 开放和插件机制。

### 2.2 MVP 非目标

为保证前三个月可交付，以下能力不进入 MVP 首版：

- AI 故障预测、智能推荐、混合云集成
- 复杂可视化工作流设计器
- Kubernetes 高可用部署和跨数据中心容灾
- 110+ OS 完整适配
- 细粒度多租户配额和企业版许可证体系
- 插件市场和完整商业化能力

这些能力保留在 Phase 2 或 Phase 3，避免 MVP 范围失控。

## 3. 目标用户与关键场景

### 3.1 目标用户

- **数据中心运维团队**: 批量上线、巡检、维修和退役物理服务器
- **私有云基础设施团队**: 管理 OpenStack、Kubernetes、VMware、Proxmox 等底层裸金属节点
- **硬件测试与交付团队**: 做服务器验收、Burn-in 测试、硬件信息采集和 BOM 输出
- **平台管理员**: 维护镜像、模板、权限、审计、告警和系统配置

### 3.2 关键业务场景

1. 新服务器通电后自动 PXE 启动，平台发现设备并采集硬件信息。
2. 运维人员在 CMDB 中补充归属、位置、BMC、用途和标签。
3. 运维人员选择镜像、网络、RAID、分区和安装模板，发起部署工作流。
4. 平台通过 iPXE、HTTP、Metadata API 和自动化安装模板完成 OS 安装。
5. 服务器安装完成后自动注册，进入运行监控和告警管理。
6. 故障发生时，平台通过告警、日志、WebSSH、BMC 和物理定位辅助处理。
7. 服务器退役时，平台记录退役原因、执行数据擦除流程并保留审计记录。

## 4. 设计原则

- **先闭环，后扩展**: MVP 优先跑通发现、录入、部署、监控、审计的主链路。
- **单机可部署，架构可演进**: 首版用 Docker Compose 降低部署门槛，后续平滑演进到 Kubernetes。
- **服务边界清晰**: 启动、部署、资产、监控、BMC、运维工具、认证审计拆分为清晰模块。
- **危险操作可追溯**: 电源控制、固件升级、密码修改、批量脚本、数据擦除必须有确认和审计。
- **兼容现网环境**: 支持 ProxyDHCP、多网段和不接管现有 DHCP 的部署模式。
- **插件化适配硬件差异**: RAID、BMC、网卡、GPU、磁盘定位等厂商差异通过适配器扩展。

## 6. 架构决策记录

### ADR-001: 后端使用 Go + Gin + GORM

**决策**: 后端首版使用 Go、Gin 和 GORM。

**原因**:

- Go 适合网络服务、并发任务、系统命令调用和静态编译部署
- Gin 轻量、生态成熟，适合 API Gateway 和业务服务
- GORM 能快速支撑 MVP 的 PostgreSQL 数据模型和迁移

**替代方案**: Echo、Fiber、Python FastAPI。MVP 不并列保留多个 Web 框架，避免团队决策成本。

### ADR-002: 前端使用 React + Ant Design + Vite

**决策**: 前端首版使用 React、Ant Design、Vite、ECharts、Xterm.js。

**原因**:

- Ant Design 更适合资产列表、表单、向导、权限、审计等企业后台场景
- React 生态对复杂状态、WebSocket、终端组件和可视化扩展支持成熟
- Vite 降低开发和构建复杂度

### ADR-003: MVP 使用 PostgreSQL + Redis + 本地镜像库

**决策**: MVP 只强依赖 PostgreSQL 和 Redis，镜像先落本地文件系统。

**原因**:

- PostgreSQL 负责资产、任务、审计、配置、用户和权限数据
- Redis 负责缓存、会话、轻量队列和部署状态
- 本地镜像库降低部署门槛，MinIO 延后到 Phase 2

### ADR-004: 监控首版不强依赖 InfluxDB

**决策**: MVP 监控指标可先写 PostgreSQL 或简化保留 7 天，Phase 2 再引入 InfluxDB。

**原因**: 监控体系容易放大工程量。MVP 只需证明采集、展示、告警链路可用。

### ADR-005: 首版采用单体服务分模块，后续再微服务拆分

**决策**: MVP 后端可以是一个 Go 主服务内的多模块结构，而不是一开始拆成多个微服务。

**原因**: 裸金属平台外部依赖复杂，过早微服务化会增加部署、调试和事务一致性成本。模块边界在代码中保留，Phase 2/3 再按压力和团队规模拆分。

## 7. 核心功能模块设计

### 7.1 智能网络启动引擎

**灵感来源**: Tinkerbell + iVentoy

**职责**: 提供 PXE/iPXE 启动、启动菜单、启动脚本生成和安装环境引导。

**MVP 能力**:

- 支持 Legacy BIOS 和 UEFI 的 PXE/iPXE 引导
- 支持 ProxyDHCP 和内置 DHCP 二选一配置
- 提供 TFTP 和 HTTP 文件服务
- 根据服务器 MAC、架构和部署任务动态生成 iPXE 脚本
- 启动日志、启动次数、最近启动时间可追踪

**Phase 2+ 能力**:

- ISO 直接启动和 HTTP Range 访问优化
- 多网段并行服务
- 启动菜单目录映射
- ISO 级别和系统级启动密码保护
- 软链接镜像库和镜像去重

**关键风险**:

- 与现有 DHCP 冲突导致生产网络异常
- UEFI、Secure Boot、不同厂商网卡固件行为差异
- TFTP 性能不足导致大规模并发启动失败

**控制措施**:

- 默认推荐 ProxyDHCP，内置 DHCP 需显式启用
- 每个网段独立配置，启动前执行 DHCP 冲突检测
- 大文件统一走 HTTP，TFTP 只承载启动加载器和最小脚本

### 7.2 资产与配置管理中心

**灵感来源**: HrnrKitOS + Tinkerbell

**职责**: 建立服务器资产台账，管理生命周期、硬件信息、位置、租户、标签和状态历史。

**MVP 能力**:

- 手动录入服务器和批量导入基础资产
- PXE 发现新设备并按 MAC 生成待认领资产
- 采集 CPU、内存、磁盘、网卡、主板、序列号等基础硬件信息
- 生命周期状态: `discovered`、`ready`、`deploying`、`running`、`maintenance`、`retired`、`scrapped`
- 标签、机房、机柜、U 位、负责人、租户字段
- 状态变更历史和审计日志

**Phase 2+ 能力**:

- BOM 清单导出 Excel/PDF
- RAID/Bond/分区可视化规划
- 保修期限、供应商、采购批次管理
- 设备重新启用、批量属性修改、租户批量分配

**关键风险**:

- MAC 地址重复或多网卡导致设备误匹配
- 自动发现信息不完整，影响后续部署

**控制措施**:

- 设备指纹使用 MAC + 序列号 + 主板 UUID 组合
- 自动发现资产进入待确认状态，不直接进入可部署状态

### 7.3 多 OS 自动化安装

**灵感来源**: iVentoy + HrnrKitOS + Tinkerbell

**职责**: 管理 OS 安装模板、变量渲染、自动化安装文件和部署过程追踪。

**MVP 能力**:

- 支持 3 种 Linux 安装模板: Ubuntu Server、Rocky Linux、Debian
- 支持 Kickstart、Preseed/Autoinstall、Cloud-init 基础能力
- 支持主机名、IP、网关、DNS、SSH 公钥、root 密码哈希变量注入
- 支持安装后脚本回传执行状态
- 实时展示部署阶段、部署日志和失败原因

**Phase 2+ 能力**:

- Windows Unattend.xml、WinPE 和驱动注入
- VMware ESXi、Proxmox、openEuler、Kylin、UOS 支持
- 文件注入、驱动库、角色模板、加密分区、复杂网络 Bond/VLAN
- 安装过程截图和调试模式

**关键风险**:

- 不同发行版安装参数差异大
- 自动分区可能误清数据

**控制措施**:

- 安装前必须展示磁盘擦除确认
- 模板按发行版和版本维护，不使用一个模板覆盖所有系统
- 安装任务必须绑定目标设备，防止误装

### 7.4 工作流编排引擎

**灵感来源**: Tinkerbell Workflow Engine

**职责**: 统一部署、硬件采集、测试、维护和退役的步骤编排。

**MVP 能力**:

- 顺序执行工作流
- 支持脚本任务、HTTP 任务、等待任务、人工确认任务
- 任务状态持久化，支持失败重试和取消
- 工作流执行日志和步骤耗时统计
- 内置标准 Linux 安装工作流

**Phase 2+ 能力**:

- 条件分支、并行任务、循环、可视化设计器
- 工作流版本管理和回滚
- A/B 测试和审批节点

**可靠性要求**:

- 每个步骤应具备幂等性设计
- 工作流失败后可从失败点继续或重新执行
- 所有步骤输出必须进入日志和审计链路

### 7.5 BMC 带外管理

**灵感来源**: Tinkerbell Rufio + HrnrKitOS

**职责**: 统一 Redfish、IPMI 和 Intel AMT 的带外管理能力。

**MVP 能力**:

- BMC 凭据加密保存
- Redfish/IPMI 电源状态查询
- 单台和批量开机、关机、重启
- BMC 连通性测试
- 所有电源操作写入审计日志

**Phase 2+ 能力**:

- Serial over LAN、KVM 跳转、Web 控制台临时凭据
- BMC 密码自动轮换
- 固件版本查询和谨慎升级
- 恢复出厂设置

**安全要求**:

- 批量关机、重启、固件升级必须二次确认
- BMC 密码只允许密文存储，不在日志中输出
- 权限应区分查看、控制、配置和固件管理

### 7.6 全栈硬件监控与自愈

**灵感来源**: HrnrKitOS + Tinkerbell

**职责**: 采集运行指标、硬件健康状态和日志，并提供告警与有限自愈。

**MVP 能力**:

- SSH Agentless 采集 CPU、内存、磁盘、网络和基础进程状态
- 支持可选 Agent 注册，但不作为首版强依赖
- 基础告警规则: 离线、磁盘满、CPU 高、内存高、磁盘 SMART 异常
- 告警列表、确认、关闭和处理记录
- 指标保留 7 天

**Phase 2+ 能力**:

- ECC、RAID、GPU、风扇、电源、温度、链路状态
- Agent 热升级
- 自愈策略: 服务重启、Agent 重启、IPMI 重启、链路切换建议
- 物理磁盘定位和机箱定位灯
- 监控大屏、热力图和趋势分析

**关键风险**:

- SSH 凭据管理复杂
- Agentless 高频采集对目标服务器和平台都有压力

**控制措施**:

- 默认低频采集，关键指标按需提高频率
- SSH 凭据使用凭据池和权限隔离
- 自愈策略默认只告警建议，自动执行需单独授权

### 7.7 运维工具套件

**灵感来源**: HrnrKitOS

**MVP 能力**:

- WebSSH 终端和会话审计
- 批量脚本执行，支持并发限制和超时
- 一键采集系统日志、dmesg、基础硬件日志
- 执行结果汇总和失败机器列表

**Phase 2+ 能力**:

- SFTP 文件传输
- RDP 跳转和 Web RDP
- RAID/GPU 专项日志采集
- CPU、内存、磁盘 IO、网络性能测试

### 7.8 元数据服务

**灵感来源**: Tinkerbell Hegel

**MVP 能力**:

- 提供 Cloud-init 兼容的基础 Metadata API
- 通过 MAC、IP 或部署任务 ID 匹配服务器
- 输出主机名、网络配置、SSH 公钥、用户数据和安装脚本变量
- 记录每次 metadata 访问日志

**API 端点示例**:

- `GET /metadata/instance-id`
- `GET /metadata/hostname`
- `GET /metadata/network`
- `GET /metadata/ssh-keys`
- `GET /userdata`

**安全要求**:

- Metadata API 默认只对部署网段开放
- 用户数据中不得返回明文密码
- 元数据访问应绑定部署任务上下文

### 7.9 镜像管理中心

**灵感来源**: iVentoy + Tinkerbell

**MVP 能力**:

- ISO 上传、删除、下载、启停用
- SHA256 校验和文件大小记录
- 镜像分类、架构、操作系统、版本、标签
- 镜像测试状态: 未测试、测试通过、测试失败
- 本地文件系统镜像库

**Phase 2+ 能力**:

- MinIO 对象存储
- 远程同步、缓存、去重、软链接、冷热分层
- ISO 内部文件 HTTP 直接访问优化

### 7.10 网络拓扑管理

**MVP 能力**:

- 维护管理网、部署网、业务网的网段配置
- 网关、DNS、VLAN、DHCP/ProxyDHCP 配置记录
- 部署前网络连通性测试

**Phase 2+ 能力**:

- 网络拓扑可视化
- 交换机 LLDP/SNMP 自动发现
- VLAN 隔离验证和网络性能监控

## 9. 核心业务流程

### 9.1 新设备发现流程

1. 服务器通电并进入 PXE。
2. DHCP/ProxyDHCP 捕获 MAC、架构和启动模式。
3. 平台生成 discovery iPXE 脚本。
4. 设备进入临时发现环境。
5. 发现环境采集硬件信息并上报。
6. 平台基于 MAC、序列号、主板 UUID 做去重。
7. 新设备进入 `discovered` 状态，等待运维确认。

### 9.2 裸金属部署流程

1. 运维人员选择目标服务器、镜像、安装模板、网络和磁盘策略。
2. 平台执行部署前检查: BMC 连通性、网段配置、镜像校验、目标状态、危险操作确认。
3. 平台创建 Deployment 和 WorkflowRun。
4. BMC 重启服务器并设置 PXE 启动。
5. iPXE 请求动态脚本，进入安装环境。
6. Metadata API 下发主机名、网络、用户数据和安装脚本。
7. OS 自动安装并执行 post install。
8. 服务器重启进入新系统并回调平台注册。
9. Deployment 变为 `success`，服务器状态变为 `running`。

### 9.3 监控告警流程

1. 采集器按周期通过 SSH 或 Agent 获取指标。
2. 指标写入存储层。
3. 告警规则计算指标阈值和状态变化。
4. 触发 Alert 并发送通知。
5. 运维人员确认、处理、关闭告警。
6. 处理过程写入审计和维护记录。

### 9.4 BMC 电源操作流程

1. 用户选择服务器并发起电源操作。
2. API Gateway 校验 RBAC 权限。
3. 高风险操作要求二次确认。
4. BMC 服务解密凭据并调用 Redfish/IPMI。
5. 平台记录操作结果、耗时和审计日志。

### 9.5 镜像导入流程

1. 用户上传 ISO 或登记本地镜像路径。
2. 平台计算 SHA256 和文件大小。
3. 用户填写 OS、版本、架构、标签和说明。
4. 镜像状态为 `enabled`，测试状态为 `untested`。
5. 成功完成一次安装后可标记为 `tested_passed`。

## 10. API 设计草案

### 10.1 资产 API

- `GET /api/v1/servers`: 查询服务器列表
- `POST /api/v1/servers`: 新建服务器资产

- `PATCH /api/v1/servers/{id}`: 修改服务器属性
- `POST /api/v1/servers/{id}/retire`: 退役服务器
- `GET /api/v1/servers/{id}/inventory`: 查询硬件采集信息

### 10.2 部署 API

- `POST /api/v1/deployments`: 创建部署任务
- `GET /api/v1/deployments`: 查询部署任务列表
- `GET /api/v1/deployments/{id}`: 查询部署详情
- `POST /api/v1/deployments/{id}/cancel`: 取消部署
- `GET /api/v1/deployments/{id}/logs`: 查询部署日志

### 10.3 镜像 API

- `POST /api/v1/images`: 上传或登记镜像
- `GET /api/v1/images`: 查询镜像列表
- `PATCH /api/v1/images/{id}`: 修改镜像元数据
- `DELETE /api/v1/images/{id}`: 删除镜像
- `POST /api/v1/images/{id}/verify`: 重新校验镜像

### 10.4 BMC API

- `GET /api/v1/servers/{id}/bmc/power`: 查询电源状态
- `POST /api/v1/servers/{id}/bmc/power-on`: 开机
- `POST /api/v1/servers/{id}/bmc/power-off`: 关机
- `POST /api/v1/servers/{id}/bmc/reboot`: 重启
- `POST /api/v1/servers/{id}/bmc/check`: BMC 连通性检查

### 10.5 监控与告警 API

### 10.6 审计 API

- `GET /api/v1/audit-logs`: 查询审计日志
- `GET /api/v1/audit-logs/{id}`: 查询审计详情

## 11. 安全设计

### 11.1 认证与鉴权

- MVP 支持本地用户、密码登录和基础 RBAC。
- Phase 2 支持 LDAP/AD/OAuth2/SAML。
- 权限按资源和操作拆分，例如资产查看、部署执行、BMC 控制、镜像管理、系统配置。

### 11.2 凭据管理

- BMC 密码、SSH 密钥、系统安装密码必须加密存储。
- 日志和审计中不得输出明文凭据。
- 密码字段 API 返回时只显示脱敏状态。
- Phase 2 可接入 KMS 或 Vault。

### 11.3 网络隔离

- 管理网、部署网、业务网建议物理或 VLAN 隔离。
- Metadata API 默认只允许部署网段访问。
- DHCP 服务启用前必须确认服务模式和网段。

### 11.4 危险操作保护

以下操作必须具备二次确认、权限校验和审计记录：

- 批量关机、批量重启、批量重装
- 磁盘擦除、重新分区、数据销毁
- BMC 密码修改和固件升级
- 批量脚本执行
- 删除镜像、删除模板、删除资产

**功能范围**:

- 基础 PXE/iPXE 网络启动，支持 BIOS 和 UEFI
- ProxyDHCP 和内置 DHCP 二选一

- 设备自动发现和硬件基础采集

- 安装成功后服务器状态自动变为 `running`。

### 14.5 安全与审计

- 用户登录、资产变更、部署、BMC 操作、批量脚本均有审计记录。
- 未授权用户不能执行部署和 BMC 操作。
- 高风险操作必须二次确认。

### 14.6 部署与运维

- Docker Compose 能在目标 Linux 主机完成启动。
- 平台重启后资产、任务、审计和配置数据不丢失。
- 基础健康检查接口可用。

## 16. 关键技术挑战与应对

### 16.1 多架构支持

**挑战**: x86_64、ARM64、Legacy BIOS、UEFI 的启动路径、内核、驱动和安装参数差异明显。

**应对**:

- 按架构管理启动文件、内核和 initramfs
- iPXE 模板按架构和启动模式选择
- 硬件发现阶段记录架构并影响后续镜像筛选

### 16.2 大规模并发部署

**挑战**: 同时部署数百台服务器时，HTTP 带宽、数据库写入和任务调度会成为瓶颈。

**应对**:

- MVP 加入并发限制和分批部署
- 大文件通过 HTTP，保留 Range 请求能力
- Phase 2 增加多节点镜像缓存和任务队列

### 16.3 硬件兼容性

**挑战**: RAID、BMC、网卡、GPU、磁盘定位强依赖厂商差异。

**应对**:

- 设计厂商适配器接口
- MVP 优先支持标准 Redfish/IPMI 和通用 Linux 工具
- 建立兼容性数据库，记录已测试设备型号和固件版本

### 16.4 安全性

**挑战**: 平台持有 BMC、SSH、系统安装等高价值凭据，且能执行关机、重装、擦盘等高风险操作。

**应对**:

- 凭据加密、审计、RBAC、二次确认作为 MVP 必需项
- 管理网和部署网隔离
- 日志脱敏和敏感字段访问控制

### 16.5 工作流可靠性

**挑战**: 部署过程跨网络、BMC、目标机器、安装系统和平台服务，失败点多。

**应对**:

- 任务状态持久化
- 每个步骤记录输入、输出、日志和错误
- 支持取消、重试和从失败点继续
- 关键步骤设计为幂等

## 19. 风险与里程碑

### 19.1 主要风险

- MVP 范围过大导致三个月无法交付
- PXE/DHCP 配置不当影响现网
- 自动安装误擦生产服务器数据
- 硬件兼容性测试不足
- 凭据管理和审计设计不到位

### 19.2 风险缓解

- 严格按 Phase 1 范围执行
- 默认 ProxyDHCP，内置 DHCP 需显式启用
- 部署目标必须绑定资产 ID 和 MAC，擦盘必须二次确认
- 建立硬件兼容性清单
- MVP 即引入 RBAC、审计和凭据加密

### 19.3 里程碑

| 里程碑 | 时间 | 交付物 |
|--------|------|--------|
| M1 | 第 1 个月 | API 框架、CMDB、镜像管理、PXE 基础服务 |
| M2 | 第 2 个月 | Linux 自动安装、Metadata API、顺序工作流 |
| M3 | 第 3 个月 | BMC、基础监控、告警、审计、Docker Compose 演示环境 |
